# 監査レポート — 本番公開前 コスト/セキュリティ点検

> 担当：財務 / コストリスク ／ 日付：2026-06-14 ／ 対象：公開直前の `main`
> 観点：**「予期せぬ高額請求」をゼロにできているか**（オーナー最大の懸念）

---

## 結論（先に）

- **API キー漏洩リスク → 現状おおむね安全 ✅**（git にもフロントにも実キーは無い）
- **コスト暴走リスク → 🔴 No-Go。公開前に止血必須**
  サービスが完全公開なのに **ジョブ投入の上限が事実上ゼロ**。
  悪意ある1人、あるいはバズによる殺到で **Claude 課金が青天井**になりうる。

オーナーが最も恐れている「とんでもない額の請求」は、**鍵の流出より「無制限ジョブ投入」から来る**。

---

## A. API キー漏洩リスク（懸念①）→ 合格 ✅

| 項目 | 結果 |
|------|------|
| `.env` が git 追跡外 | ✅ `.gitignore` で `web/backend/.env` 等を除外。追跡は `*.example` のみ |
| フロントに機密混入 | ✅ `ANTHROPIC_API_KEY` / R2 / Resend をフロントソースが一切参照せず |
| ブラウザ公開変数 | ✅ `NEXT_PUBLIC_API_BASE`（API の URL）のみ。これは公開されて正常 |
| 通信の暗号化 | ✅ nginx で TLS + HSTS |

**残る注意点（中）**：Claude キーは Hetzner VPS の `web/backend/.env` に平文で存在する。
サーバーが乗っ取られれば流出する。→ SSH 鍵認証のみ・ファイアウォール・最小権限で守る（USER_TASKS の範囲）。

---

## B. コスト暴走リスク（懸念②）→ 🔴 重大

### B-1. レート制限が存在しない【最重要・止血必須】
`main.py` の `create_job` は **同時処理 10 本**を超えたときだけ 429 を返す。
これは「同時実行数」の制限であって「投入総量」の制限ではない。

- IP 単位の制限 … **なし**
- メール単位の制限 … **なし**
- 全体の日次上限 … **なし**

→ 攻撃者はキューが空くたびに投げ続けられる。**1日のジョブ数に上限が無い = Claude 課金に上限が無い。**

### B-2. 入力動画の尺に上限がない【止血必須】
`fetch` は動画尺をチェックせず丸ごとDL→全文字起こし→**その全文を Claude に投入**。
`analyze.py` は出力こそ `max_tokens=3000` だが、**入力（文字起こし全文）は無制限**。
→ 10時間配信を1本投げるだけで、単発でも巨大な入力トークン課金が発生する。

### B-3. Anthropic Console のスペンドリミット未設定【オーナー必須】
コードのバグや想定漏れをすべて貫通されても、**Console のハードキャップが最後の砦**。
これが無いと理論最大コストは∞。**コードでは設定できない。オーナーが Console で設定する必要がある。**

### B-4. 副次的な濫用経路（中）
- nginx に `limit_req`（プロキシ層レート制限）がない。
- 第三者のメールアドレスでジョブ投入可能 → 被害者に「完成しました」メールが届くスパム経路（Resend 通数も消費）。

---

## C. 良かった点（過剰防御は不要）

- ✅ SSRF 対策：URL を `youtube.com` / `youtu.be` に正規表現で限定（フロント zod + バック pydantic の二重）
- ✅ 署名 URL は 7 日失効、期限切れ R2 オブジェクトは Beat タスクで自動削除
- ✅ ジョブ ID は UUIDv4（推測不可）。認証なし GET の事実上のトークンとして妥当
- ✅ CORS は `FRONTEND_ORIGIN` + `*.vercel.app` に限定

---

## D. 是正タスク（優先度順）

### 🔴 公開前（止血 / No-Go 解除条件）
1. **Anthropic Console で月次スペンドリミット設定**（オーナー作業・最優先）
2. **レート制限の実装**：IP単位・メール単位・全体日次の上限 → 超過は 429（ENG）
3. **入力動画の尺上限**：`config.yaml` に上限秒を追加し、`fetch` 後に超過なら failed（ENG）
4. **Claude 投入文字起こしの長さキャップ**：尺上限の二重化（ENG）

### 🟡 公開後すぐ
5. nginx `limit_req` でプロキシ層の粗いレート制限
6. メール単位の投入頻度制限（スパム経路の緩和）
7. 1ジョブの実コスト（トークン数・処理秒）をログ化 → 理論最大コストを実額で再計算

---

## E. 理論最大コストの考え方

```
1日の最大コスト = 1日の最大ジョブ数 × 1ジョブの最大コスト
```
- 現状：両因子とも上限なし → **理論最大コスト = ∞**（= No-Go の理由）
- 是正後：`(日次ジョブ上限) × (尺上限から決まる1ジョブ上限)` で**事前に金額が確定**できる。
- この確定額をオーナーが「1日あたり許容額」として承認できれば Go。
