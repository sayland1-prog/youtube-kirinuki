## 3. 依存関係マップ（何がブロックするか）

```
マーケ:ポジショニング ──▶ デザイン:LP/コピー ──▶ デザイン:LP最終
ENG:API契約(§4) ◀──握る──▶ デザイン:enum表示対応表        （P0-P1で相互確定）
        │                          │
        └──────────┬───────────────┘
                   ▼ (両方ブロッキング)
        ENG-FE実装 / ENG-BE実装 (P2)
                   ▼
              E2E疎通 (P2) ──▶ デプロイ(P3) ──▶ オーナー公開判断
ENG:実コスト計測 ──▶ マーケ:料金プラン案 (P3)
マーケ:計測イベント要件 ──▶ ENG:計測実装 (P2-P3)
```

### 並行可能（同時に走らせてよい）
- **SP1**：マーケ「ポジショニング」／デザイン「フロー骨子」／ENG「API契約・DB・pipelineラッパー化」は**相互独立で並行**。
- ENG の `pipeline.py` ラッパー化（`pipeline_adapter`）はデザイン待ち不要＝**先行着手**してよい。

### 直列（先に終わらせないと進めない）
- フロント実装は **デザイン画面仕様 ＋ API契約** の確定後（P2）。
- LP最終ビジュアルは **マーケのコピー** 確定後。

## 6. リスクと対応

| リスク | 影響 | 対応 |
|--------|------|------|
| 処理が15-30分→待ち時間離脱 | CVR低下 | デザイン「閉じてOK・メール通知」UXを最優先。SP2で設計 |
| YouTube仕様変更でDL失敗 | 中核機能停止 | ENG：yt-dlp最新追従＋失敗リトライ＋失敗メール（既存CLAUDE.md準拠） |
| 著作権トラブル | レピュテーション/法務 | 自己責任モデルを規約明記。マーケ文言で無断転載を煽らない |
| API契約のブレ | フロント/バック手戻り | 契約を§4に一元化。変更時は3者通知を必須化 |
| コスト超過 | 採算 | ENGが1本あたりコスト計測 → マーケ料金設計に反映 |
