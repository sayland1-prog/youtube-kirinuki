## 4. オーケストレーションの基本原則

1. **価値の流れに沿って順序づける**
   `マーケ（誰に何を）` → `デザイン（どう見せる）` → `エンジニア（どう作る）`。
   ただし一方通行ではなく、各下流からのフィードバックを上流へ戻す（例：技術制約でUXを調整、UXの発見でメッセージを磨く）。

2. **単一の正（Single Source of Truth）を守る**
   プロダクト要件＝このファイル。担当間で食い違ったら、ここを基準に裁定する。重複ドキュメントを作らせない。

3. **完全自動・最小操作の思想を全担当に徹底させる**
   このプロダクトの核は「URLとメールを入れるだけ」。マーケの訴求もUXもこの一点に収束させる。機能や画面を増やしたくなったら、まず「入れるだけ」を薄めないか問う。

4. **MVP を死守する**
   MVPスコープ（§2）外の提案が出たら歓迎しつつ「フェーズ2以降」のバックログへ。MVPを肥大させない。

5. **並行作業を設計する**
   デザインとマーケの初動は並行可能。エンジニアはデザイン確定を待つ部分（UI）と先行できる部分（バックエンド基盤・パイプラインのラッパー化）を切り分けて走らせる。

## 6. 担当への依頼の出し方（委任プロトコル）

担当に仕事を振るときは、毎回この形式でブリーフを渡す（曖昧な丸投げをしない）：

```
【依頼先】 どの担当か
【背景】   なぜ今これをやるか（North Star のどこに効くか）
【成果物】 何を・どんな形式で出してほしいか
【制約】   守るべき要件・前提（§2 から関連分を抜粋）
【完了条件】 何を満たせば完了とみなすか
【依存】   先行して必要な他担当の成果物、後続でこれを使う担当
```

担当からの成果物は**受け取って終わりにせず**、North Star との整合・他担当への影響を確認してから次へ繋ぐ。

## 8. 品質基準（Definition of Done）

プロダクト全体として「完成」とみなす条件：

- [ ] 完全公開URLで、ログインなしに URL＋メール入力 → 送信ができる
- [ ] 利用規約（著作権は自己責任）への同意なしには送信できない
- [ ] 進捗がリアルタイムに見える（downloading→transcribing→analyzing→clipping→done）
- [ ] 完了時：画面DLボタン表示 ＋ メール送付（有効期限1週間の署名URL）
- [ ] 失敗時：画面エラー表示 ＋ 失敗メール送付
- [ ] 同時10本まで処理、超過はキュー待ちと分かる表示
- [ ] 既存 `pipeline.py` / `scripts/` を壊していない（CLIとしても従来通り動く）

## 9. 守るべき制約（全担当に徹底）

- **既存パイプラインを壊さない**：`pipeline.py`・`scripts/`・`config.yaml` はWebサービスの土台。CLI利用も維持する。変更は最小限・後方互換で。
- **著作権はシステムで保証しない**：許可チェックはせず、利用規約への同意でユーザー自己責任とする。マーケ・UIの文言もこの前提を踏み外さない。
- **小規模最適**：月100ユーザー規模。過剰な分散・冗長構成は作らない。VPS1台＋マネージドサービスで回す。
- **「入れるだけ」を薄めない**：機能追加は常にこの体験を守れるか審査する。
